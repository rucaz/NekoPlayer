## 3. 竞品优势拆解

### 3.1 值得借鉴的功能

1. **歌词系统**
   - LRC 解析与同步
   - 逐字/逐行高亮
   - 动效（缩放/渐变）
   - 桌面歌词悬浮窗

2. **本地音乐管理**
   - 自动扫描本地音频
   - 元数据自动匹配
   - 封面/歌词自动获取

3. **推荐算法**
   - 日推歌单
   - 私人FM (随机播放+跳过学习)

4. **内容维度扩展**
   - 专辑详情页
   - 艺人主页
   - 评论区浏览

5. **听歌识曲**
   - 麦克风音频指纹匹配

### 3.2 技术亮点

- **Electron 优势**: 快速迭代、Web生态丰富
- **内存优化**: v0.9.3 专门做了内存回收优化
- **打包优化**: 修复了体积过大的问题

## 4. 我们的优势

### 4.1 差异化定位

| 维度 | 我们胜出 |
|------|---------|
| **音源特色** | B站二次元音乐 + 咪咕，避开网易云红海 |
| **跨平台** | KMP一套代码，iOS/Android同步 |
| **移动端** | 竞品只有Windows，移动端空白 |
| **体积** | 原生应用，比Electron更轻量 |

### 4.2 技术架构优势

- Kotlin Multiplatform 原生性能
- Compose Multiplatform UI一致性
- SQLDelight 类型安全数据库

## 5. 优化升级建议

### 5.1 P0 优先级 (核心功能补齐)

#### ① 歌词系统
```
需求: 显示LRC歌词，支持滚动同步
实现思路:
- 接入歌词API (B站部分视频有字幕，咪咕有歌词)
- 本地LRC解析器
- NowPlayingScreen底部扩展歌词区域
- 长按歌词可跳转
```

#### ② 本地音乐扫描
```
需求: 扫描设备本地音频文件
实现思路:
- Android: MediaStore API
- iOS: MPMediaQuery
- 添加到播放队列/歌单
```

#### ③ 播放历史
```
需求: 记录播放过的歌曲
实现思路:
- History表 (song_id, played_at)
- "最近播放"歌单自动维护
```

### 5.2 P1 优先级 (体验提升)

#### ④ 歌单/歌曲详情页
```
竞品参考: 网易云专辑页、艺人页
内容: 歌曲列表 + 基本信息 + 相似推荐
```

#### ⑤ 歌词动效
```
参考: 竞品动效歌词
实现: Compose Canvas绘制逐字高亮
```

#### ⑥ 睡眠定时
```
需求: 设置播放停止时间
实现: 系统定时器 + 暂停播放
```

### 5.3 P2 优先级 (差异化功能)

#### ⑦ 听歌识曲
```
难度: 高 (需要音频指纹服务)
替代方案: 接入第三方API (ACRCloud)
```

#### ⑧ 桌面端扩展
```
思路1: 用Compose Desktop复用代码
思路2: 等待JetBrains官方桌面支持
```

#### ⑨ 插件系统
```
参考: 竞品的桌面插件
移动替代: 小组件 (Widget)
```

## 7. 总结

### 竞品给我们的启示

1. **桌面端 Electron 虽快但重** - 我们走原生性能路线是对的
2. **内容深度很重要** - 不只是播放，还有歌词、详情、推荐
3. **本地+云端结合** - 竞品本地音乐管理做得很好
4. **视觉体验是护城河** - 动效歌词、主题自定义增加粘性

### 我们的机会点

1. **移动优先** - 竞品只有Windows，移动端是空白市场
2. **差异化音源** - B站二次元音乐有忠实受众
3. **技术栈先进** - KMP跨平台比Electron更现代化
4. **体积优势** - 原生应用启动快、占用低

### 一句话战略

> 守住"移动端+二次元音源"基本盘，逐步补齐歌词/本地音乐/推荐等核心体验，未来可考虑Compose Desktop进军桌面端。
